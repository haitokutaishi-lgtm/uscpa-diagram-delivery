# 図解1テーマ — 完成の定義・テンプレ（配信用HTML）

低価法HTML（`publish-html/inventory-lcm-reading-drill.html`）を標準とする。**Discord 配信〜URL の枠組みは既存のまま**、ここでは **中身の必須要件・分割ルール・ファクトチェック・著作権上の取り扱い**を固定する。

---

## 方針（合意事項）

| 項目 | 内容 |
|------|------|
| 参照範囲 | **教材の論点はすべて参照対象**。ただし **難解論点を優先的に図解**する。 |
| バックログ | **図解適性**を最重視。ほかの並べ方は柔軟でよい。 |
| アウトプット | **解説・構成は参照しつつ、文章・MC・数値シナリオはオリジナル執筆**。第三者教材の**転載・長文の複製はしない**（下記「著作権・安全運用」）。 |
| 品質 | **ファクトチェックは必須**。特に **MC の正解・計算・用語**は公開前に入念に確認する。 |

## 1配信のスコープと分割ルール（運用デフォルト）

**基本: 1配信（1 HTML）＝ 受講生の頭の中で「1本のレール」**

- **1レールの例**: ひとつの分岐軸（FIFO/LIFO の読み分け）、ひとつの手順（DTA の型）、ひとつのタイムライン（銀行調整の帳簿側／銀行側）。
- **同一HTMLに収める**: 図解の核心が **同じレールの連続ステップ**（例: LCM で market 算出 → cost との比較までを同一テーマとみなすかは、その回の学習目標で決める。迷うときは **「market まで」「在庫評価額まで」でテーマIDを分ける**）。
- **分割を推奨する条件**（いずれか該当で検討）:
  - **別ASC・別サブトピック**で、用語の対応関係が薄い（例: 連結ののれんと棚卸未実現を1枚に詰め込むと回路が二重になる）。
  - **スマホで一読15分超**になりそうな情報量（「まず覚える3つ」が機能しなくなる）。
  - **MC の検証軸が複数**になり、理解確認が「どちらの論点か」曖昧になる。

分割した場合は **テーマIDを分け、レジストリに別行、`schedule` / `diagram_urls` も別URL** で揃える。

## HTML の必須構成（完成の定義）

公開・配信可能とみなす前に、次をすべて満たす。

### 構造（10部構成・2026-07改訂）

FAR論点解説シリーズ（S5 連結・内部取引消去）の構成を取り込み、spec 生成テンプレ（`ops/generate_diagram_from_spec.py`）は次の順で出力する。

1. **ヘッダ**: テーマを一文で明示。参照元があっても **転載ではないこと**を必要に応じて短く記載（例: 予備校・問題集の転載なし）。
2. **目次**: アンカー付き（番号は自動採番）。
3. **① 30秒でわかるこのページ**: 核心3点（spec の `three`）を**冒頭に**置く。読者はまずここだけで要点を持ち帰れる。
4. **② 先に言葉をそろえる**: 用語ミニ辞書（`glossary`・**3語まで**）。本文前に用語の意味を固定する。
5. **③ よくある悩み**: 受講生×コーチの対話（誤解①〜③）。**diagram-delivery/SETUP.txt** のペルソナ・アバター規約に従う。
6. **④ 会計の背景と仕訳** / **⑤ 計算の型**: 図・表・読む順番（Lucide、絵文字禁止）。
7. **⑥ 対話で整理**: 発展・境界ケースの対話。
8. **⑦ 横串マップ**（`crossmap`）: 他論点（別ASC・前後のステップ・他科目）とのつながりを2〜3件。「単独論点で終わらせない」ための必須セクション。
9. **⑧ 出題パターンの引き出し**（`patterns`）: 「MC の出方」2〜3パターン＋「解く手順は毎回同じ○ステップ」。出題側の視点を先に見せる。
10. **⑨ 理解確認 MC（必須）**: 下記「MC の要件」。
11. **⑩ まとめ**: 「**1分で人に説明できる形**」（`one_minute_html`・1段落）＋「**今日のチェックリスト**」（`checklist`・自己テスト形式 4〜5項目）。
12. **フッタ**: オリジナル執筆・テンプレ起因の注記。

### 理解確認 MC の要件（毎回）

| 要件 | 内容 |
|------|------|
| **掲載** | **毎回、必ず**「理解確認」セクションを設ける（`id="mc"` 推奨）。 |
| **最低本数** | **2問**: **問題1＝計算またはケース適用**、**問題2＝概念・英文の読み分け**（低価法HTMLと同型）。論点が概念のみのテーマでは、**情境の異なる概念問題2問＋計算がない理由を解説で明示**するなど、**理解の軸が二重に検証できる**組み合わせにする。 |
| **ここで見るポイント** | 各問の英文の**前**に、「ここで見るポイント（読む前に）」（`focus`）を1〜2点。何と何を混同させに来る問題かを先に示す。 |
| **誤答の作られ方** | 解説内に「誤答の作られ方」（`distractors_html`）を必ず入れる。**各誤答肢が「どういうミスをした人向けに作られたか」**を1行ずつ分解する（単なる「Bは誤り」では不可）。 |
| **言語** | ストップ・選択肢は **試験で使う英語**。解説は **日本語でフル**（`<details>` の中に段階的に記載）。 |
| **オリジナル** | **数値・会社名・事実関係はオリジナル**。**Released MC・商用問題集・第三者教材からのコピペ禁止**（類題として再構成するのは可）。 |
| **冒頭の注記** | MCブロック直後に、**ファクトチェックを実施したこと**と **照合した ASC（主題番号）** を短文で書く（低価法ページの注記を参照）。 |

## ファクトチェック（品質保証・必須）

**自己レビューのみで公開しない。** 次のいずれかを実施し、レジストリに記録する。

1. **別コンテキストでのレビュー**: メイン作成と別スレッド／別担当で、**Codification（該当 ASC）に照らして**  
   - 図解本文の断言（ルール・例外・境界）  
   - **各MCの正解・計算過程・誤答肢の妥当性**  
   を確認する。  
2. **人間による再確認**: エージェントが使えない場合は、**公式 Codification または社内で承認された二次資料**で同項目を確認する。

### チェックリスト（テーマごとに検証）

- [ ] **適用する ASC（サブトピックまで）**を特定し、URLまたは参照箇所をメモした。
- [ ] 図解の**「まず覚える3つ」**に反する例外・更新がないか確認した。
- [ ] **MC 各問**: 正解の根拠を ASC上の考え方で説明できる。計算問は**再計算**した。
- [ ] **いずれかの誤答肢**が「正解に見える」場合、その理由が解説で潰せているか確認した。
- [ ] 税率・金額・定義は**設問文と矛盾**していない。
- [ ] 「試験では」「典型的には」等の言い切りは、**過度な一般化**になっていない（必要なら注釈）。

## 著作権・安全運用（法的リスクを抑える）

これは**法的助言ではない**。境界事例は**権利者への確認または専門家への相談**を推奨する。

### 必ず避ける

- 第三者の教材・問題集・スライドからの**長文コピー・軽微な改変の転載**。
- **AICPA / Wiley 等のリリース問題そのもの**や、**記憶に基づく再現**であっても**実題に近似しすぎる**英文・選択肢の複製。
- 教材PDFや書籍の**図表・表のそのままスクリーンショット貼付**。

### 推奨する

- **論点・順序・注意点は「参照」**し、**語句は自分の言葉で書き直す**。
- MCは**オリジナルのファクトパターン**で新規作成（数値・名称・シナリオを変える）。
- 参照したい場合は **ASC・FASB の公開テキスト**（利用条件に従う）や、**自組織が権限を持つ資料**に寄せる。
- フッタ等に **「予備校・問題集の転載なし」「オリジナル」** を明記する（既存HTMLと同様）。

### クレジット（参照の表明）

- 内部運用で「aki教材の◯章を参照」とレジストリに書くのは可。**外部公開ページに第三者の商標・教材名を出す必要はない**が、方針で必要なら**最小限の参照表明**にとどめる（長い引用は避ける）。
